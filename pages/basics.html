Basics

Android Framework

Android is an event based framework that queues up events similar to other event frameworks. Android provides call backs for these different events allowing different objects to be notified when certain events occure. Android apps are native apps and thus are tied very closley with the android operating system. The operating system gets events such as touches, networking data received, as well as communication from other processes and queues them up for the app to process.

Context

Android is built around the concept of a context. A context is a way to represent the current application environment. We will talk a little more about these later but the important part is to understand what they are. Essentially, anytime the app needs to interact with the environment, things like the filesystem or screen, then you need a context in which to interact. The root context is the application context which is held by the application singleton. However each activity and service also holds on the context as well.

Event Loop

We have mentioned several times that the android framework has an event loop. If your not familier with the concept the idea is that the operating system will queue up events and then the main thread will process them. Events can be a list of things but mainly they are things related to the android lifecycle, more on this later, and user actions. For example when the user presses a button an touch event will get triggered and the approptiate callback will get called on the view that it happened on. Another great example is when the phone gets low battery. A low battery event will get triggered and the application will process it.

Now that we have started with some basic android lets talk about the three main important pieces of an android application.


1.) Application

The application object holds a lot of power. Not only does it control the event loop and acts as the root of the application but because it is a singleton you are able to access it from anywhere. The application object owns the activities as well as the services and anything that seems hidden from view. The nice thing that most of it is hidden and works behind the scenes but it is still good to know where all of that lives.


The application object is normally where developers store the state of the application because it gives the rest of the application access to it. Many people overload the default applicatoin object by subclassing it and overriding its members. This is a basic stub of the application object that you would need to overload. Many of the members you can leave but the nice thing is that the event loop will call each one of these methods on different events. 


public class OurApplication extends Application {
    @Override
    public void onCreate() {
        super.onCreate();
    }

    @Override
    public void onTerminate() {
        super.onTerminate();
    }

    @Override
    public void onTrimMemory(int level) {
        super.onTrimMemory(level);
    }

    @Override
    public void onConfigurationChanged(Configuration newConfig) {
        super.onConfigurationChanged(newConfig);
    }

    @Override
    public void onLowMemory() {
        super.onLowMemory();
    }

    public OurApplication() {
        super();
    }

    @Override
    public void registerComponentCallbacks(ComponentCallbacks callback) {
        super.registerComponentCallbacks(callback);
    }

    @Override
    public void unregisterComponentCallbacks(ComponentCallbacks callback) {
        super.unregisterComponentCallbacks(callback);
    }

    @Override
    public void registerActivityLifecycleCallbacks(ActivityLifecycleCallbacks callback) {
        super.registerActivityLifecycleCallbacks(callback);
    }

    @Override
    public void registerOnProvideAssistDataListener(OnProvideAssistDataListener callback) {
        super.registerOnProvideAssistDataListener(callback);
    }

    @Override
    public void unregisterOnProvideAssistDataListener(OnProvideAssistDataListener callback) {
        super.unregisterOnProvideAssistDataListener(callback);
    }

    @Override
    public void unregisterActivityLifecycleCallbacks(ActivityLifecycleCallbacks callback) {
        super.unregisterActivityLifecycleCallbacks(callback);
    }
}

The application object is where all events trickle down from so it has access to all the possible call backs. Many applications though only have a gui and thus putting information in the application object become a little overkill. The solution to this is activities.


2.) Activities

An activity is probably the single most important type of class that developers will be interacting with when creating android apps. They are so important in fact that were dedicating a whole page to it. Here we will give a little preview into how they work. Activities control the current gui. A common way to develop apps is to create an activity for each page and then change activities when needed. Below is an example for how the main activity looks after creation. 

 
public class OurActivity extends Activity {

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main_activity2);
    }


    @Override
    public boolean onCreateOptionsMenu(Menu menu) {
        // Inflate the menu; this adds items to the action bar if it is present.
        getMenuInflater().inflate(R.menu.menu_main_activity2, menu);
        return true;
    }

    @Override
    public boolean onOptionsItemSelected(MenuItem item) {
        // Handle action bar item clicks here. The action bar will
        // automatically handle clicks on the Home/Up button, so long
        // as you specify a parent activity in AndroidManifest.xml.
        int id = item.getItemId();

        //noinspection SimplifiableIfStatement
        if (id == R.id.action_settings) {
            return true;
        }

        return super.onOptionsItemSelected(item);
    }
}

Activites control a context just like the application object. This simple fact makes them extremely powerful. They are able to process events off of the event loop. When the application gets a new event one of the first things it does is check what callback he is supposed to call. Activities will get call backs for all of androids main events allowing them to keep track of what is going on with the system.


We will talk a little more about what each piece of this does on the next page. For now lets jump to the final important piece to applications.


3.) Services

Services are like gui less activities. Essentially, they are able to process events directly from the application just like activities but they don't need to keep track of gui. Everything in android that we have talked about so far happens on the main thread. That means that if you need to do time consuming things in an activity then your going to block your gui. The solution to this was the introduction of services. Services are still actually processed on the main thread but they provide a place where you can put another thread and allow the service to offload the work. Code for the services are very simple and have the simmilar callbacks to activities, the application, and other context classes. A shell service looks like:


public class MyService extends Service {
    public MyService() {
    }

    @Override
    public IBinder onBind(Intent intent) {
        // TODO: Return the communication channel to the service.
        throw new UnsupportedOperationException("Not yet implemented");
    }
}

We will dive more into the binder callbacks in the services section but essentially they provide a way for the service to communicate back to the activities, whether they be in another app or in this one.



So we have established that activites and services both receive events using the main threads event loop. They also communicate to each other via these events allowing for things to be queued up really nicley. The powerful thing about these events is that anyone on the phone is able to send them allowing other applications to interact with the services or activities in your app. To do this android created the concept of intents.

Intents

Intents are what android uses as events and to queue up an event you need to send an intent via a context. When you send the intent. The android os will process the event and then give it to whatever applications have subscribed to it. This is extremly powerful because it provieds a very easy way to do inter process communication between applications on the phone.
